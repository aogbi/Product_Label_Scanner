01 Project Overview

Product Label Scanner is an enterprise-grade Django REST application that automates the extraction of critical compliance metadata — specifically LOT/batch numbers and expiry dates — from pharmaceutical product label images. It targets a real problem: manual label verification is slow, error-prone, and a regulatory liability.

The system ingests raw label images and returns clean, structured JSON records ready for inventory or compliance databases. The core challenge is that real pharmaceutical labels are visually noisy and OCR output is fragmented — this pipeline was built explicitly to handle that reality.

Tech Stack

Backend Framework
Django 6.0 + Django REST Framework
OCR Engine
EasyOCR (PyTorch-based neural network)
Date Parsing
Custom regex + dateparser (fallback only)
Database
SQLite (dev) → PostgreSQL (production)
Frontend
Vanilla HTML / CSS / JavaScript (SPA)
Core Logic
ocr_service.py — 474 lines, 7-step pipeline

What it does (user flow)

User uploads a label image → EasyOCR extracts all text bounding boxes → spatial grouping reconstructs lines → regex extracts LOT number → multi-strategy parsing extracts expiry date → result saved to database and returned as JSON. Manual correction via PATCH /api/labels/<id>/ keeps humans in the loop.

03 OCR Pipeline — 7 Steps

The entire extraction logic lives in ocr_service.py. Each step handles a specific class of noise or ambiguity inherent in pharmaceutical label images.

01

EasyOCR Text Extraction

Runs the PyTorch model over the image. Returns a list of (bounding_box, text, confidence) tuples. Each bounding box has 4 corner coordinates. Model is loaded once at server boot via lazy global init — not per-request.

02

Spatial Grouping (Y-axis proximity)

Blocks whose left_y values fall within ±30 pixels are merged into the same logical line, then sorted left-to-right by X-coordinate. This reconstructs the label's actual reading layout from disconnected fragments. Without this, "10 / 20 2 6" is four separate unrelated tokens — with it, they become "EXP: 10/2026".

03

LOT Number Extraction

Collapses single-character spaces ("L 0 T" → "LOT"), then applies regex with a negative lookahead to avoid matching keywords like EXP: as lot values. A rejection set blocks pharmaceutical units (EXP, DOM, MFG, etc.) from being extracted.

# Pattern with negative lookahead
r"(?:LOT|BATCH|LoT)[#:\s]*([A-Z0-9][\w\-\.]{1,30})(?!\s*:)"
04

Line Cleaning

Collapses fragmented digits ("20 2 6" → "2026"), replaces - and . between digits with /, strips spaces around slashes. Produces normalized strings ready for date logic.

05

Date Strategy Selection

Branches on content type: if a month name is found (JAN, FEB, FÉV...) → delegate to dateparser.search_dates(). If purely numeric → process word-by-word: slash-delimited tokens go to validate_slash_date(), pure digit strings go to process_pure_number_word().

06

Safe Parsing (_safe_parse)

Tries strict regex formats in order before falling back to the dateparser library. This prevents the day/month swap bug.

# Priority order:
1. MM/YYYY  →  08/2025
2. MM/YY    →  07/25  →  07/2025
3. DD/MM/YYYY
4. DD/MM/YY
5. dateparser (text months only)
07

Expiry Selection (Latest-Date Heuristic)

All valid dates found across the label are collected. The latest chronological date is selected as the expiry. This correctly handles labels that show both a manufacture date (DOM/FAB) and an expiry date (EXP/PER) — the manufacturing date is always older.

06 Challenges & Solutions

📦
Challenge 1: Fragmented OCR Text
Problem
EasyOCR doesn't read text as sentences — it returns disconnected bounding boxes. A date like 10/2026 becomes ["10", "20", "2", "6"] at arbitrary positions. Standard regex is useless against this.
Solution
Custom Spatial Grouping Algorithm: group blocks sharing similar Y-axis values (±30px), then sort by X-coordinate within each group to reconstruct natural reading order — turning fragments into full lines.
📋
Challenge 2: Two-Column Label Merging
Problem
Labels with LOT on the left and EXP on the right get spatially merged into one line: "LOT: EXP: Rn6 2 0 10 20 2 6" — making it hard to know which value belongs to which field.
Solution
The regex handles it: negative lookahead (?!\s*:) skips any token followed by a colon (i.e., a keyword), and the rejection set blocks EXP, DOM, MFG from ever being returned as lot values.
🔢
Challenge 3: Ambiguous 6-Digit Coded Dates
Problem
Manufacturers compress dates to save space: 102026 could be MMYYYY (Oct 2026) or YYYYMM (nonsense). No delimiter to hint at the format.
Solution
Deterministic parse_6_digits(): test both interpretations. If first two digits ≤ 12 and last four are 2020–2035 → it's MMYYYY. If reversed → YYYYMM. Rejects anything else.
📅
Challenge 4: dateparser Day/Month Swap Bug
Problem
dateparser.parse("08/2025") sometimes returns August 20, 2025 instead of August 2025 — swapping month and day unpredictably depending on internal locale settings.
Solution
_safe_parse() wrapper tries explicit regex formats first (MM/YYYY, MM/YY, DD/MM/YYYY). dateparser is only called as a last resort for text-based month names like JUN 2026 or FÉVRIER 2025.
🏷️
Challenge 5: False Positive Dates (Noisy Labels)
Problem
Pharmaceutical labels contain barcodes (03400927708022), dosages (1,9mg), and random product codes (971019) that naive date parsers match as valid dates.
Solution
Multi-layer filtering: skip numbers attached to units (mg/ml/kg), bound years strictly to 2020–2035, strip GTIN/barcodes (5+ digits) before dateparser sees them, and apply keyword proximity weighting to EXP, PER, USE BY.
⚡
Challenge 6: AI Model Init Overhead
Problem
EasyOCR was originally instantiated inside the API view — meaning every image upload re-loaded the full PyTorch model into memory, adding 3–5 seconds of cold-start latency per request.
Solution
Lazy global initialization via get_reader(): the easyocr.Reader is instantiated once at server boot and reused for all requests. Eliminates per-request model loading entirely.

07 Deployment Postmortem

Moving from local development to Railway's cloud environment surfaced two significant infrastructure issues. Both are documented here as architectural learnings.

Issue 1 — 502 Bad Gateway (OOM Crash)

# Railway logs during image scan
GET    /                 200    3s       # Server up, frontend loads fine
GET    /api/history/     200    289ms    # Database healthy
POST   /api/scan/        502    17s      # CRASH — OOM kill

Root Cause

PyTorch model weights consume ~200–300MB RAM on import alone. Railway Free Tier caps containers at 500MB. When a scan is triggered, the memory spike kills the Gunicorn worker — the proxy gets no response, returns 502. Reducing workers to 1 via --workers 1 wasn't enough; the model itself is too large for the container.

Production Fix

Migrate from self-hosted EasyOCR to a managed cloud inference API (Google Cloud Vision or AWS Textract). The Django application sends the image, the cloud API returns text, and the custom Python parsing pipeline runs on the lightweight response. Zero memory spike.

Issue 2 — 404 on Media Files

GET    /media/labels/med3.jpeg    404    285ms

Root Cause

Setting DEBUG = False in production causes Django to stop serving /media/ files by design — a WSGI app should not stream binary files directly. This is a security principle, not a bug.

Fix

Short-term: set DEBUG = True in Railway environment variables. Long-term: route uploaded media to Amazon S3 or Google Cloud Storage and serve via CDN. The Django app stores only the URL reference.

Production Architecture Roadmap

☁️

Cloud OCR APIs

Replace EasyOCR with Google Cloud Vision or AWS Textract. Pay per scan (~$0.001), eliminate RAM ceiling entirely.

⚙️

Async Processing

Decouple scanning from the web request using Redis + Celery. User gets immediate response; WebSockets push result when ready.

🗄️

Cloud Storage

Move /media/ to Amazon S3. Serve via CloudFront CDN. Django stores URLs, not files.

Key lesson: Deep learning inference cannot run synchronously inside a web request loop at scale. The custom text pipeline (spatial grouping, date parsing, regex extraction) remains 100% intact under any of these upgrades — only the OCR source changes.

08 API Reference

POST /api/scan/ Upload image and trigger OCR pipeline

Request: multipart/form-data with field image

{
  "id": 1,
  "image": "/media/labels/med1.jpeg",
  "raw_text": "LOT: Rn6 2 0 EXP: 10 20 2 6 DOM: 2 0 2 3",
  "lot_number": "Rn620",
  "expiry_date": "10/2026",
  "expiry_date_parsed": "2026-10-01",  // ISO 8601 for DB storage
  "confidence": 0.7234,
  "created_at": "2026-05-21T12:00:00Z"
}
Returns 201 Created on success.
GET /api/history/ List all scans, newest first
Returns paginated array of all ProductLabel records. Used by the frontend history grid.
PATCH /api/labels/<id>/ Manual correction of OCR results

Allows human-in-the-loop correction when OCR fails. Request: application/json

{
  "lot_number": "RN620",
  "expiry_date": "10/2026"
}
DELETE /api/labels/<id>/ Remove scan record and associated image file
Deletes both the database record and the stored image from /media/labels/.
