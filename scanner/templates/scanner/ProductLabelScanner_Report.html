01 Project Overview

Product Label Scanner is a Django application that automates extraction of expiry dates from pharmaceutical product label images. Manual label verification is slow and error-prone; this pipeline turns a photo of a label into a structured, machine-readable expiry date.

The hard part is not reading text — it is reading broken text. EasyOCR returns disconnected bounding boxes, and on real labels a date like 10/2026 is frequently shattered into fragments such as "10", "20", "2", "6", sometimes with stray letters glued on ("Rn6"). The pipeline is built specifically to recover a correct date from that mess, or to honestly return nothing when no date is recoverable.

User Flow

Image upload → EasyOCR extracts text boxes → boxes grouped into lines by vertical proximity → every parseable date is collected → the largest (latest) valid date is selected as the expiry → result returned as JSON and stored. Manual correction via PATCH /api/labels/<id>/ keeps a human in the loop.

03 Extraction Pipeline

Every stage targets a specific class of noise in pharmaceutical label images. The pipeline is fully deterministic — dateparser is only consulted for textual month names.

01

EasyOCR Text Extraction

Runs the PyTorch model and returns (bounding_box, text, confidence) tuples. Boxes below a 0.4 confidence threshold are dropped. The reader is loaded once at boot via a lazy global, never per-request.

02

Spatial Line Reconstruction

Boxes are grouped by comparing their vertical centre, with tolerance scaled to box height (the larger of ~20px or 60% of the box height). This keeps big and small text grouping correctly and tolerates slightly skewed labels, then sorts each line left-to-right. Disconnected fragments become a readable line.

03

Normalisation & Junk Rejection

Collapses spaced single digits ("1 0 2 0 2 6" → "102026"), converts - . , between digits to /, and tidies spaces around slashes. Long digit runs that aren't clean 6-digit date blobs (barcodes, GTINs, lot codes like 2673846) are flagged as junk and skipped.

04

Token Parsing

Each token is parsed by a single resolver handling every common format: MM/YYYY, MM/YY, YYYY/MM, DD/MM/YYYY, plus 6-digit MMYYYY/YYYYMM and 4-digit MMYY blobs. The governing rule: in any pair, the value ≤ 12 is the month; the other is the year. Bare years alone are rejected.

# the month is always the value <= 12
(6, 2027)  → 2027-06
(2027, 6)  → 2027-06
(6, 27)    → 2027-06
(27, 6)    → 2027-06
05

Scrambled-Digit Salvage

When a date field is shattered beyond clean parsing ("Rn6 2 0 10 20 2 6"), all digits in the field are concatenated and a sliding window hunts for a valid MM+YYYY pattern around any 4-digit year, tolerating leading/trailing garbage. Salvaged dates are tagged reconstructed so the selector can treat them with suspicion.

06

Candidate Collection

Every valid date across all lines is gathered, each bounded to the 2020–2035 year window. Textual month-name lines (JUN 2026, FÉVRIER 2025) are routed to dateparser after long barcode runs are stripped out.

07

Expiry Selection (Largest-Date Rule)

Labels are not used to decide. Every valid date is compared and the latest one wins. The only cleanup: a reconstructed date that merely duplicates a date already parsed cleanly is discarded as OCR noise, so garbage cannot invent a larger fake date.

// 04

04 Selection Logic

The selection rule went through several iterations before settling on its current form. The history is worth recording because each version fixed a real bug and exposed the next one.

Evolution of the rule

①
Label-priority (EXP wins)

Prefer any date tagged EXP, never return a FAB/DOM date. Fixed the original "manufacture date returned as expiry" bug, but failed when OCR scrambled the EXP field into garbage while the DOM field stayed clean.

②
EXP-must-be-after-DOM floor

Use the manufacture date as a hard floor and reject any candidate not strictly later. Conceptually clean, but too strict — it returned nothing on labels where the only recoverable date equalled the manufacture date, and the "EXP after DOM" assumption is not guaranteed in practice.

③
Largest valid date (current)

Parse every date, compare, take the largest. Labels are ignored for the decision. The single safeguard: drop reconstructed dates that duplicate a cleanly-parsed date, killing scrambled-OCR noise without inventing values. Simple, predictable, and matches how the dates actually relate on a label.

def select_expiry(candidates):
    # parse every date, then take the latest one
    clean = {c['date'] for c in candidates if not c.get('reconstructed')}
    usable = [c for c in candidates
              if not (c.get('reconstructed') and c['date'] in clean)]
    usable.sort(key=lambda c: c['date'])
    return usable[-1]   # largest (latest) date wins
Design trade-off: because the rule always takes the largest date, a label whose manufacture date is genuinely later than its expiry would yield the manufacture date. This is rare and was an explicit, accepted choice — "compare the two dates and take the larger one" — in exchange for predictable behaviour on the common case.

05 Challenges & Solutions

📦
Challenge 1: Fragmented OCR Text
Problem
EasyOCR returns disconnected boxes, not sentences. A date like 10/2026 arrives as ["10", "20", "2", "6"] at arbitrary positions.
Solution
Spatial line reconstruction groups boxes by vertical centre with height-scaled tolerance, then sorts by X to rebuild reading order — turning fragments into a coherent line.
🔢
Challenge 2: Ambiguous Coded Dates
Problem
Manufacturers compress dates with no delimiter: 102026 could be MMYYYY (Oct 2026) or YYYYMM.
Solution
Deterministic resolvers test both readings using the month-≤-12 rule and the 2020–2035 year bound. Anything that fits neither is rejected rather than guessed.
🧩
Challenge 3: Scrambled Digit Soup
Problem
Real OCR produces fields like "Rn6 2 0 10 20 2 6" — letters fused to digits, digits chunked wrong. No single token parses.
Solution
find_date_in_digit_blob() concatenates all digits and slides a window around any 4-digit year, grabbing the adjacent ≤-12 month and ignoring surrounding garbage. The result is flagged reconstructed.
📅
Challenge 4: dateparser Day/Month Swap
Problem
dateparser.parse("08/2025") can return August 20 instead of August 2025, swapping fields by locale.
Solution
Explicit deterministic resolvers run first for all numeric formats. dateparser is only invoked for textual month names like JUN 2026.
🏷️
Challenge 5: False-Positive Dates
Problem
Barcodes (03400927708022), dosages (1,9mg) and product codes match as dates under naive parsing.
Solution
Junk filtering rejects long non-date digit runs, bounds years to 2020–2035, strips 5+ digit runs before dateparser, and skips unit-attached numbers.
⚡
Challenge 6: Model Init Overhead
Problem
Instantiating EasyOCR per request reloaded the PyTorch model every upload, adding 3–5s cold-start latency.
Solution
Lazy global init via get_reader(): the reader is built once and reused for every request.

06 Deployment Postmortem

Moving from local development to a cloud container surfaced two infrastructure issues, recorded here as architectural learnings.

Issue 1 — 502 Bad Gateway (OOM)

# logs during an image scan
GET    /                 200    3s       # frontend loads fine
GET    /api/history/     200    289ms    # database healthy
POST   /api/scan/        502    17s      # CRASH — OOM kill

Root Cause

PyTorch weights consume ~200–300MB on import. A free-tier 500MB container can't absorb the memory spike when a scan runs; the worker is killed and the proxy returns 502. Reducing to a single worker wasn't enough — the model itself is too large for the container.

Production Fix

Move inference to a managed cloud OCR API (Google Cloud Vision / AWS Textract). Django sends the image, the API returns text, and the custom parsing pipeline runs on the lightweight response — no memory spike.

Issue 2 — 404 on Media Files

GET    /media/labels/med3.jpeg    404    285ms

Root Cause

With DEBUG = False, Django stops serving /media/ by design — a WSGI app shouldn't stream binary files directly. A security principle, not a bug.

Fix

Route uploaded media to object storage (S3 / GCS) and serve via CDN; Django stores only the URL reference.

Production Roadmap

☁️

Cloud OCR APIs

Replace EasyOCR with a managed vision API. Pay per scan, drop the RAM ceiling entirely.

⚙️

Async Processing

Decouple scanning from the request with Redis + Celery; push results when ready.

🗄️

Cloud Storage

Move /media/ to S3 + CDN. Django stores URLs, not files.

Key lesson: deep-learning inference can't run synchronously inside a web request at scale. The parsing pipeline — spatial grouping, date resolution, junk rejection, selection — stays 100% intact under any of these upgrades; only the OCR source changes.

08 Known Limitations

Honest boundaries of the current build, kept visible so they aren't mistaken for bugs.

"___"
LOT stubbed in scan_label
2035
Hard upper year bound
∅
Returns empty on unreadable input

LOT extraction is disabled. The extract_lot_number() function and its patterns still exist, but scan_label() hardcodes lot_number = "___". Re-enable by wiring the function back into the return value.

The year window is fixed at 2020–2035. Any date outside that range is rejected. This is a deliberate guard against barcode false positives, but it must be widened before 2035 or for products with longer shelf lives.

Truly corrupted OCR is unrecoverable. When digits are merged wrong at the source (e.g. "10" and "20" read as units instead of "1 0" / "2 0"), no parser can reliably reconstruct the intended date. The right fix there is image quality and the OCR confidence threshold, not more parsing logic.

Largest-date selection ignores labels. See §04 — by design, the latest date always wins, so a label with a manufacture date later than its expiry would return the wrong field. Accepted trade-off for predictability.

Bottom line: the pipeline is tuned to be honest. When it can recover a date it does; when the input is genuinely unreadable it returns nothing rather than a confident guess — the correct behaviour for a compliance system.
